逆向工程：
“最低层”抽象：能够导出过程的设计表示文档
“低层”抽象：能够导出程序和数据的结构信息
“中层”抽象：能够导出数据流和控制流模型
“高层”抽象：实体关系模型

层次越高，完备性越低

鱼骨图（Ishikawa图）是一种发现问题“根本原因”的方法，通常用来进行因果分析
为了得到工作分解结构（Work Breakdown Structure,WBS）最底层的交付物，必须执行一系列的活动，对这些活动的识别及归档的过程就是活动定义。

V模型：测试模型、测试全程介入、测试提前



软件重用：横向重用（不同领域）和纵向重用（一类具有较多公共性的应用领域之间进行软部件重用）
计算机不同应用领域也算不同领域（数据结构、横向应用和纵向应用），标准函数是一种典型的、原始的横向重用机制

边界值分析的取值：
刚好等于，刚好大于、刚好小于
例如10<X<30,则取值X=9,X=10,X=30,X=31进行边界值分析


恢复测试：监测系统的容错能力
强度测试：测试系统在异常情况下的承受能力，检测系统在极限状态运行时，性能下降的幅度是否在允许的范围内
性能测试：检测系统是否满足系统设计方案说明书对性能的要求
可靠性测试：用平均失效间隔时间（MTBF，Mean Time Between Failures）和因故障停机时间(MTTR，Mean Time To Repairs)来衡量




OOA（面向对象分析）包含3个活动：建模系统功能、发现并确定业务对象、组织对象并发现对象间的关系
功能测试主要采取黑盒测试的方法


隐式调用：一个事件的触发导致了另一个模块中的过程调用
将显示调用策略替换为隐式调用策略能够提高系统的灵活性，但会降低系统的性能

二层C/S架构：客户端直接访问服务器数据库
三层C/S架构：比二层C/S架构增加了一个应用服务器
三层C/S架构分为数据层、功能层、表示层

面向服务的架构（SOA，Service Oriented Architecture)


无状态的消息：服务使用者向服务提供者发送的每条消息都必须包含全部信息
有状态的消息：发送的信息依赖于服务使用者与服务提供者共享的特定环境信息
等幂消息：向软件代理发送多条重复消息与发送单条消息效果相同
SOA:面向服务的架构
SOA的生命周期：建模、组装、部署、管理→(迭代)→建模(所有步骤都连接着控制)

Web Sevice是技术规范，SOA是设计原则

ABSD(Architecture-Based Software Design, ABSD)基于体系结构的软件设计，强调由商业、质量和功能需求组合驱动软件架构设计，采用视角和视图来描述软件架构，采用用例和质量场景来描述需求（用例描述的是功能需求，质量属性场景描述的是质量需求，或侧重于非功能需求）
ABSD有三个基础：
对系统进行功能分解、采用架构风格实现质量属性与商业需求、采用软件模板设计软件结构
在ABSD的活动中：
架构复审活动的目的是识别潜在风险，及早发现架构设计中的缺陷和错误
架构演化活动针对用户需求的变化，修改应用架构，满足新的要求


软件架构的评估方式：
1.基于调查问卷（或检查表）的评估方式
2.基于场景的评估方式
3.基于量度的评估方式

敏感点（Sensitivity Point）：一个或多个构件（和/或构件之间的关系）特性，是实现一个特定质量属性的关键特性
权衡点（Tradeoff Point）：影响多个质量属性的特性，是多个质量属性的敏感点

风险点与非风险点不是以标准专业术语出现的，只是一个常规概念，即可能引起风险的因素，称为风险点。

ATAM（Architecture Tradeoff Analysis Method）：评价软件架构的一种综合全面的方法，主要对软件体系结构的设计结果进行评估，由于评估是在详细设计、实现和测试之前的工作，所以不涉及系统的实现代码和测试。
ATAM不是一种精确的评估方法，主要形式是评审会议。
ATAM的四个活动阶段（注意不是步骤）：
需求收集（场景和需求收集）
架构视图描述（架构视图和场景实现）
属性模型构造和分析
架构决策与折中
（carat）

整个评估过程强调以 属性 作为架构评估的核心概念
通常采用效用树对质量属性的描述进行刻画与排序
ATAM框架主要关注系统的需求说明
ATAM主要在系统开发之前，针对性能、可用性、安全性和可修改性等质量属性进行评价和折中。

合成（Composite）模式组合多个对象形成树形结构以表示整体-部分的结构层次

装饰器模式：动态地给一个对象增加其它职责
适配器模式：将一个接口转换成客户想要的另一个接口
模板方法：定义一个操作中算法骨架，将一些步骤暂缓到子类中实现


备忘录（Memmento）模式：在不破坏封装的前提下，捕获一个对象的内部的状态，并在该对象之外保存这个状态，从而可以在以后将该对象恢复到原先保存的状态

调停者（Mediator）模式定义一个对象封装一系列多个对象如何相互作用。

中介者模式：用一个中介对象来封装一系列的对象交互，它使各对象不需要显式的相互调用，从而达到低耦合，还可以独立地改变对象间的交互

MVC:数据模型主要指实体对象的数据保存（持续化）

层次系统组织成一个层次结构，每一层为上层服务，并作为下层用户。允许将一个问题分解成一个增量步骤序列的实现。层次系统最广泛的应用是分层通信协议。
管道/过滤器风格的软件体系结构：每个构件都有一组输入和输出，过滤器不知道上游和下游的标识，特别重要的过滤器必须是独立的实体，不能与其它的过滤器共享数据

管道—过滤器的架构风格支持分阶段数据处理

IDL（Interface Definition Language）接口定义语言
POA(Portable Object Adapter)可携带对象适配器，是一个引导客户端的请求到具体对象的应用机制

断点和它的监视机制是一种回调机制，属于隐式调用的架构风格

采用闭环结构的软件通常由几个协作构件共同构成，且其中主要构件彼此分开，能够进行替换与重用，但是闭环结构通常适合于简单任务（如机器装配等），并不适用于复杂任务。
分层结构的特点是引入抽象层，在较低层次不确定的实现细节在较高层次会变得确定，并能够组织层间构件的协作，系统结构更加清晰。
管道—过滤器：支持分阶段的数据处理

DSSA（Domain-Specific Software Architecture）特定领域软件体系架构。领域设计活动的主要目的是为了获取DSSA，该活动参加人员中，领域专家的主要任务是提供关于领域中系统的需求规约和实现的知识。
DSSA通常是一个具有三个层次的系统模型：
领域开发环境、领域特定应用开发环境、应用执行环境，应用工程师主要在领域特定应用开发环境工作。
参与DSSA的人员可以划分为4种角色：
领域专家、领域分析师、领域设计人员、领域实现人员
DSSA的基本活动：
领域分析：主要目的是获取领域模型，领域模型描述系统之间共同的需求，即领域需求
领域设计：主要目的是获取DSSA，DSSA描述领域模型中表示需求的解决方案
领域实现：主要目标是依据领域模型和DSSA，开发和组织可重用信息，并对基础软件架构进行实现。

提高软件架构的质量属性：
提高可用性：Ping/Echo、心跳机制、命令/响应机制、异常机制、冗余机制
提高安全性：身份认证、限制访问、检测攻击、维护完整性（入侵检测、用户认证、用户授权、追踪审计）
提高可修改性：运行时注册、接口-实现分离、信息隐藏（注意这个）
提高可靠性：主动冗余
提高性能：队列调度、引入并发、维持数据或计算多个副本、增加可用资源、控制采样病毒、限制执行时间、固定优先级调度
提高可测试性：记录-回放


在0.5秒内得到响应：性能
在0.5小时内恢复正常：可用性
二次开发要0.5年：可修改性

冗余：硬件冗余（结构冗余）、软件冗余（冗余附加）、信息冗余、时间冗余
结构冗余：静态冗余【被动冗余】、动态冗余【主动冗余】、混合冗余
信息冗余：检错码和纠错码
失效率λ与可靠度R的关系:R=e^(-λt),t为时间

程序回卷是从出错的地方重新执行程序，属于时间冗余技术。
指令复执是指重新执行出错的指令，也属于时间冗余技术。

MTTF:Mean Time To Failure
MTTR:Mean Time To Repair
可靠性的简单量度是平均故障时间间隔(MTBF)，MTBF=MTTF+MTTR
软件可用性是指在某个给定时间点上程序能够按照需求执行的概率
可用性=MTTF/(MTTF+MTTR)*100%


基准测试采用的测试程序称为基准程序，基准程序就是公认的标准程序，用它能测试多种计算机系统，比较和评价它们的性能，定期公布测试结果，供用户选购计算机参考。



MIPS(每秒百万次指令数)衡量处理机的性能
MFLOPS(每秒百万次浮点数运算)衡量向量处理机的性能

软件的可靠性：软件无故障运行的概率

排除一个错误可能会引入其它错误，所以修改错误之后要进行回归测试。

系统加速比=改进后系统的性能/改进前系统性能
系统加速比=改进前系统总执行时间/改进后系统总执行时间
